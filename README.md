# ЕГЭ Обществознание — интерактивный тренажёр

Веб-приложение для подготовки школьника к ЕГЭ по обществознанию. Цель — **85+ баллов**. Полностью на русском языке, **мятная светлая тема по умолчанию + переключатель тёмной**, адаптивная вёрстка (мобильный + десктоп).

🔗 **Демо (GitHub Pages):** https://realfactchecknews-eng.github.io/ege-obshestvoznanie/
📦 **Репозиторий:** https://github.com/realfactchecknews-eng/ege-obshestvoznanie

> **Этот README — точка входа для разработчика или ИИ-ассистента, продолжающего проект.** Ниже описаны архитектура, схемы данных, карта файлов, соглашения и пошаговые рецепты типовых доработок. Читается без знания истории проекта.

## 1. Что это и как устроено в двух словах

Это **статический клиентский SPA** (React + TypeScript + Vite + Tailwind). Вся «база» заданий, теории и терминов — это три JSON-файла, которые **бандлятся в сборку** на этапе компиляции. Прогресс ученика (ответы, серии, изученные темы, состояние карточек) хранится в **`localStorage`** — без сервера и авторизации.

Папка `/server` — **опциональный** Express + SQLite REST API. Он **не используется** развёрнутой на GitHub Pages версией (Pages раздаёт только статику и не может запускать Node). Сервер оставлен для тех, кто захочет хранить прогресс централизованно в БД. **Источник правды для данных — JSON в `client/src/data/`**, а `server/data/` — их копии.

```
Пользователь → GitHub Pages (статика из client/dist)
                 ├─ React-приложение
                 ├─ tasks.json / theory.json / terms.json (вшиты в бандл)
                 └─ localStorage (прогресс)

[опционально, локально] server/ → Express → SQLite (server/db/ege.db)
```

## 2. Технологии

| Слой | Технологии |
|------|-----------|
| Клиент | React 19, TypeScript, Vite 8, Tailwind CSS 3 |
| Состояние/навигация | Локальный React-стейт (без react-router и без внешнего стейт-менеджера). Переключение экранов — `useState<View>` в `App.tsx` |
| Хранение прогресса | `localStorage` (обёртки в `client/src/lib/storage.ts`) |
| Сервер (опционально) | Node.js (ESM), Express 4, better-sqlite3 |
| Деплой | GitHub Actions → GitHub Pages (`.github/workflows/deploy.yml`) |

Важный нюанс сборки: в `client/vite.config.ts` задан **`base: '/ege-obshestvoznanie/'`** — это нужно, потому что Pages обслуживает сайт по пути `/<repo>/`. Если репозиторий переименуют, этот `base` надо поменять, иначе ассеты не загрузятся.

## 4. Схемы данных (важно для добавления контента)

### 4.1. Задание — `client/src/data/tasks.json`

Массив объектов. Поля зависят от `type`. Поддерживаются 5 типов:

| `type` | Что это | Обязательные доп. поля | Формат `correct_answer` |
|--------|---------|------------------------|--------------------------|
| `single` | один правильный вариант | `options: string[]` | **текст** правильного варианта (для текстовых ответов добавь `"options_letters": false`) |
| `multiple` | несколько правильных | `options: string[]` | строка цифр-индексов, напр. `"134"` (порядок неважен, сверяется как множество) |
| `matching` | соответствие колонок | `match_left: string[]`, `match_right: string[]` | строка выбора для каждого левого пункта по порядку, напр. `"2132"` |
| `fill` | вписать слово | — (`options: []`) | слово/словосочетание; сверка нечувствительна к регистру, пробелам и `е/ё` |
| `written` | часть 2 (21–25), ручная проверка | `criteria: string[]` | эталонный ответ (многострочный текст); проверяется учеником вручную по `criteria` |

Полный пример (multiple):
```json
{
  "id": 61,
  "year": 2024,
  "task_number": 6,
  "type": "multiple",
  "topic": "Экономика",
  "subtopic": "Рыночная экономика",
  "question": "Выберите верные суждения о рыночной экономике.",
  "options": ["вариант 1", "вариант 2", "вариант 3", "вариант 4", "вариант 5"],
  "correct_answer": "134",
  "explanation": "Пояснение со ссылкой на статью закона, напр. «Согласно ст. 8 Конституции РФ…».",
  "source": "ФИПИ 2024"
}
```

Пример matching:
```json
{
  "id": 62, "year": 2023, "task_number": 8, "type": "matching",
  "topic": "Экономика", "subtopic": "Налоги",
  "question": "Установите соответствие между примерами и видами налогов.",
  "match_left": ["НДФЛ", "НДС", "Акциз", "Налог на имущество"],
  "match_right": ["Прямой налог", "Косвенный налог"],
  "correct_answer": "1221",
  "explanation": "…", "source": "ФИПИ"
}
```

Пример written (часть 2):
```json
{
  "id": 63, "year": 2023, "task_number": 24, "type": "written",
  "topic": "Политика", "subtopic": "Составление сложного плана",
  "question": "Составьте сложный план по теме «…».",
  "options": [],
  "correct_answer": "Примерный план:\n1. …\n2. …\n   а) …\n   б) …",
  "criteria": [
    "План содержит не менее трёх пунктов",
    "Два пункта детализированы в подпунктах",
    "Корректность формулировок"
  ],
  "explanation": "Критерий 24.1 — раскрытие темы; 24.2 — корректность. Максимум 4 балла.",
  "source": "ФИПИ, формат 2023"
}
```

Соглашения по `correct_answer`:
- `multiple` — индексы вариантов **с единицы**, в строку без разделителей (`"134"` = варианты 1, 3, 4).
- `matching` — для каждого пункта `match_left` по порядку указывается номер выбранного `match_right` (с единицы). Длина строки = числу левых пунктов.
- Логика сверки — в `client/src/lib/data.ts → checkAnswer()`. Если добавляешь новый тип задания, правь её там же.

### 4.2. Теория — `client/src/data/theory.json`

```json
[
  {
    "section": "Экономика",
    "code": "3",
    "topics": [
      {
        "id": "ec-1",
        "title": "Экономика и экономические системы",
        "text": "Многострочный текст. Переносы строк \\n отображаются как абзацы.",
        "key_terms": ["Экономика", "Факторы производства", "Рента"]
      }
    ]
  }
]
```
- `id` темы используется как ключ в localStorage для отметки «изучено» — **делай его уникальным и не меняй задним числом**, иначе у пользователя слетят отметки.
- `key_terms` рендерятся кликабельными чипами; попап подтягивает определение из `terms.json` по совпадению названия (регистр игнорируется). Если термина нет в `terms.json`, показывается заглушка.

### 4.3. Термины — `client/src/data/terms.json`

```json
[
  { "term": "Инфляция", "definition": "Устойчивый рост общего уровня цен…", "topic": "Экономика" }
]
```
Используются в попапах теории и в режиме карточек (флешкарты с интервальным повторением).

### 4.4. Разделы (`TOPICS`)

Канонический список из 6 разделов кодификатора задан **строкой** в `client/src/lib/data.ts → TOPICS`. Значение `topic` в заданиях/терминах **должно точно совпадать** с одной из этих строк, иначе задание не попадёт в фильтры и аналитику:
```
Человек и общество · Духовная культура · Экономика · Социальные отношения · Политика · Право
```

## 5. Как хранится прогресс (localStorage)

Все ключи и обёртки — в `client/src/lib/storage.ts`. Никогда не пишите в localStorage напрямую из компонентов, только через эти функции.

| Ключ | Назначение |
|------|-----------|
| `ege_answers` | массив `AnswerResult` — каждый ответ части 1 (taskId, taskNumber, topic, correct, date, ts) |
| `ege_learned` | `{ [themeId]: boolean }` — изученные темы теории |
| `ege_variants` | массив `VariantResult` — история сданных вариантов |
| `ege_flash` | состояние SRS-карточек `{ [term]: { box, due } }` (система Лейтнера, 6 «коробок») |
| `ege_streak` | `{ count, last }` — серия дней подряд; инкрементится при любом сохранённом ответе/варианте |

Аналитика (Dashboard, WeakSpots) — это агрегации `ege_answers` на лету. Сброс прогресса = удаление `ege_answers` (кнопка на дашборде).

Ключи из `PROGRESS_KEYS` (`storage.ts`) — это ровно те, что синхронизируются в облако при входе (см. §11). Функция `write()` после каждой записи шлёт событие `ege:progress-changed`, на которое подписан слой синхронизации. Тема (`ege_theme`) и сессия Supabase в синк не входят.

## 7. REST API сервера (опционально)

| Метод | Путь | Описание |
|-------|------|----------|
| GET | `/api/tasks?topic=&task_number=&limit=` | список заданий с фильтрами |
| GET | `/api/tasks/variant` | случайный вариант (по одному заданию на номер) |
| POST | `/api/progress` | сохранить результат ответа (body: `{task_id, task_number, topic, correct, date, ts}`) |
| GET | `/api/progress/stats` | агрегаты по темам и по номерам заданий |
| GET | `/api/theory` | теория из `server/data/theory.json` |
| GET | `/api/terms` | термины из `server/data/terms.json` |

> Сейчас клиент **не** ходит в это API (работает на localStorage). Чтобы переключить клиент на серверное хранение, надо заменить вызовы из `lib/storage.ts` на `fetch` к этим эндпоинтам и поднять CORS-домен. Это осознанная нереализованная развилка — см. раздел 10.

## 8. Деплой

CI описан в `.github/workflows/deploy.yml`:
1. триггер — пуш в `main` (или ручной запуск `workflow_dispatch`);
2. `npm ci` + `npm run build` в папке `client`;
3. публикация `client/dist` на GitHub Pages.

GitHub Pages включён в режиме **GitHub Actions** (Settings → Pages → Source: GitHub Actions). Никаких ручных действий после пуша не нужно — сайт обновляется автоматически за ~1–2 минуты.

## 9. Как сделать типовые доработки (рецепты)

**Добавить задание** → допиши объект в `client/src/data/tasks.json` (см. §4.1), убедись, что `topic` из списка §4.4 и `id` уникален. Пересобери (`npm run build`) или просто запусти dev. Для сервера — `npm run seed`.

**Добавить раздел теории/тему** → допиши в `theory.json`, дай теме уникальный `id`. Термины из `key_terms` желательно завести в `terms.json`.

**Добавить новый тип задания** → 1) расширь union `TaskType` в `types.ts`; 2) добавь ветку рендера в `TaskCard.tsx`; 3) добавь логику сверки в `checkAnswer()` в `lib/data.ts`.

**Поменять дневную цель / правила streak** → `DAILY_GOAL` в `Dashboard.tsx`; логика серии в `touchStreak()` в `storage.ts`.

**Поменять длительность варианта** → константа `DURATION` в `Variant.tsx`.

**Сменить палитру/тему** → цвета в `tailwind.config.js` (`theme.extend.colors`). Тёмная тема — дефолт, переключателя светлой темы пока нет.

**Переименовать репозиторий** → обнови `base` в `vite.config.ts` и все ссылки на URL Pages.

## 10. Известные ограничения и идеи для развития

- **Контента 60 заданий**. Стоит дорастить, особенно дефицитные номера: графики экономики (№10), планы (№24), мини-сочинения (№25).
- **ИИ-проверка ориентировочна** — даёт оценку по критериям, но не заменяет реального эксперта ФИПИ. Работает только при настроенном Supabase + OpenAI (см. §11).
- **Нет тестов** (ни юнит, ни e2e).
- Данные заданий вшиваются в бандл — при большом росте контента стоит вынести их в подгружаемые по сети файлы.
- Подтверждение email при регистрации зависит от настроек Supabase (по умолчанию можно отключить для мгновенного входа).

## 11. Настройка бэкенда: аккаунты + облако + ИИ (Supabase + OpenAI)

Сайт полностью работает и **без** этого раздела (офлайн-режим, прогресс в `localStorage`). Шаги ниже включают авторизацию, облачную синхронизацию и ИИ-проверку письменных заданий.

**Стек:** Supabase (auth по email+паролю + Postgres для прогресса + Edge Function как прокси к OpenAI). Ключ OpenAI хранится в секретах функции и на клиент не попадает.

### Шаг 1. Создать проект Supabase
1. Зарегистрируйтесь на [supabase.com](https://supabase.com), создайте новый проект.
2. Project Settings → API → скопируйте **Project URL** и **anon public key** (эти ключи публичные, их можно класть в клиент).

### Шаг 2. Создать таблицу прогресса
Dashboard → SQL Editor → New query → вставьте содержимое [`supabase/schema.sql`](supabase/schema.sql) → Run. Создаётся таблица `user_progress` с Row Level Security (каждый видит только свои данные).

### Шаг 3. Развернуть Edge Function для ИИ-проверки
```bash
npm i -g supabase            # CLI, если ещё нет
supabase login
supabase link --project-ref <ваш-project-ref>
supabase functions deploy check-essay
supabase secrets set OPENAI_API_KEY=sk-...     # ключ OpenAI (секрет, не в гите)
# опционально сменить модель (по умолчанию gpt-4o-mini):
supabase secrets set OPENAI_MODEL=gpt-4o-mini
```
Код функции — [`supabase/functions/check-essay/index.ts`](supabase/functions/check-essay/index.ts).

### Шаг 4. Прописать переменные окружения клиента
**Локально:** скопируйте `client/.env.example` → `client/.env.local` и подставьте URL и anon key.

**Для GitHub Pages:** добавьте те же значения в **Settings → Secrets and variables → Actions → New repository secret**:
- `VITE_SUPABASE_URL`
- `VITE_SUPABASE_ANON_KEY`

Workflow (`.github/workflows/deploy.yml`) уже пробрасывает их в сборку. После добавления секретов сделайте любой пуш в `main` — фронт пересоберётся с подключённым бэкендом.

### Как это устроено в коде
| Файл | Роль |
|------|------|
| `client/src/lib/supabase.ts` | создаёт клиент из env-переменных; `isSupabaseConfigured` гейтит всю облачную логику |
| `client/src/lib/auth.tsx` | `AuthProvider`/`useAuth`: сессия, вход/регистрация, запуск синхронизации |
| `client/src/lib/sync.ts` | merge локального и облачного прогресса (`user_progress.data` jsonb), debounce-выгрузка |
| `client/src/lib/ai.ts` | вызов Edge Function `check-essay` через `supabase.functions.invoke` |
| `client/src/components/AuthWidget.tsx` | кнопка входа/профиль в сайдбаре + модалка |
| `supabase/functions/check-essay/` | серверный прокси к OpenAI (ключ в секретах) |

Логика синхронизации: при входе локальный и облачный прогресс **сливаются** (ответы/варианты объединяются без дублей, `learned`/`flash`/`streak` — разумным слиянием), результат пишется и локально, и в облако; дальнейшие изменения уходят в облако с debounce 1.5 с.
